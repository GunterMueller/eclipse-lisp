Eclipse behavior for
Function LOAD

Description:

Filespec is merged with *default-pathname-defaults*, and the matching files are located in the file system as though with directory. If the merged result specifies a mnemonic version such as :NEWEST, the appropriate version is identified. If the merged file type is NIL, then directory will consider all file types. Links are not resolved when gathering matching files.

If more than one file matches (as, for example, if the type is NIL), then only files of canonicalized type "BIN" are considered. If there are no "BIN" files, then only files of canonicalized "LISP" type are considered. If this process yields more than one applicable file, an error of type file-error is signaled (due to unresolved wild-pathname components). If this process yields no files, the specified if-does-not-exist action is performed.

See Also:

The Character Repertoires and Interaction with Streams sections of "Characters in Eclipse", "Pathnames in Eclipse", compile-file, open.

Notes:

Note that if a specific version number is specified, but no type, load will load the binary of that version if it exists, otherwise the Lisp source. If neither a version nor a type is specified and *default-pathname-defaults* has its usual version component of :NEWEST, load will load the latest version of the file, with the binary for that version taking precedence, if it exists.

Note that the :NEWEST version of a file which has several numbered versions in the file system is the one with the highest number, regardless of file-write-date.

ANSI defines load to accept an external-format &KEY argument, but not an element-type argument. This is because loading a source file (i.e. .lisp) must handle any implementation-supported character. The Eclipse implementation of load recognizes that only base-chars can be read using the :ASCII external-format. The consequence is that when a source file is loaded and the :ASCII external-format is specified, string literals and the symbol-names of newly created symbols will be of type base-string. Otherwise, it is only specified that these will be of type string.
