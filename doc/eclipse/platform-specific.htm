Eclipse Introduction: Platform Specific Issues

UNIX

  • Except as noted, C applications linked with Eclipse on Unix systems should also link with the operating system's ld library.

Solaris

Eclipse has been compiled and tested using Solaris 2.6.
  • See chars.c for information about using the Eclipse libraries or the Eclipse executable on earlier Solaris releases.

HP-UX

Eclipse has been compiled and tested using HP-UX 9.05.
  • C applications linked with Eclipse should also link with the operating system's dld library. (This is the HP-UX name for ld.)
  • See the "doc/eclipse/UNPACKING" file for information about reading the distribution CD on systems which cannot read long file names on compact disks.

Linux

Eclipse has been compiled and tested using Redhat Linux 4.0.

WINDOWS

  • The Microsoft Windows operating system utility for obtaining elapsed CPU run time overflows after just under 50 days of continuous computer operation. A computer running Windows for this long without rebooting will report eroneous CPU run times. Thus, get-internal-run-time will not return monotonically increasing values when the values being compared straddle such an overflow. Comparisons between two such get-internal-run-time results are not valid.

  • The Eclipse-specfic function machine-id returns NIL.

NT

Eclipse has been compiled and tested using Windows-NT 4.0.

Windows

Eclipse has been tested using Windows-95.
  • Eclipse uses the Microsoft Windows Platform SDK function WaitForSingleObject() to implement listen and read-char-no-hang. WaitForSingleObject() may not be reliable on Windows-95 -- occasionally indicating that input is available when, in fact, it is not.

    Eclipse itself uses these Lisp functions only when using dribble, or when the operating system supplied standard output has been redirected to a non-interactive stream before calling clLispTopLevel. Under these circumstances, the system may sometimes appear to hang when entering expressions to the read-eval-print loop. A second #\return character entered at the keyboard will clear the problem and allow the system to accept the user input.

    Windows-NT does not appear to suffer from this problem.
