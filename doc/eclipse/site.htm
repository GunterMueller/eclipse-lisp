Eclipse Lisp: Site and Other Initial Definitions

The function short-site-name returns the value of the variable *short-site-name*.

The function long-site-name returns the value of *long-site-name* if it is non-null, otherwise it returns *short-site-name*.

The function machine-version returns the value of the variable *machine-version* if it is non-null.

The initial value of *default-pathname-defaults* is a pathname with the following components:

  • host - :unspecific
  • device - The operating-system-specific "current working device". On Unix, this is :unspecific. On Windows, it is the "current default drive" in which Eclipse was started.
  • directory - The operating-system-specific "current working directory" in which Eclipse was started.
  • name - NIL
  • type - NIL
  • version - :NEWEST

Eclipse uses the "SYS" logical pathname host during init file loading and to provide default paths for load-logical-pathname-translations and require.

The initial logical-pathname-translations for the "SYS" logical-pathname host are such that:

  • "SITE;**;" translates to '(eclipsehome "SITE" :WILD-INFERIORS) (in :COMMON :CASE) on the local machine.
  • "SOURCE;**;" translates to '(eclipsehome "SRC" :WILD-INFERIORS) (in :COMMON :CASE) on the local machine.
If the ECLIPSEHOME operating system environment variable is set, then eclipsehome, above, is the directory specified by this variable. If it is not set, then eclipsehome is the current working directory in which eclipse was started.

For example, suppose Eclipse is started on a Unix system with the current working directory /usr/eclipse and ECLIPSEHOME unset: 

  (translate-logical-pathname "SYS:SITE;FILE.LISP.2")
   => #P"/usr/eclipse/site/file.lisp.2"
  (translate-logical-pathname "SYS:SOURCE;MY-PROJECT;DEV;FILE.LISP.2")
   => #P"/usr/eclipse/src/my-project/dev/file.lisp.2"

N.B.: The "SYS" logical pathname translations can be modified by the user, for example, through an init file.

We would like to avoid the use of the ECLIPSEHOME in order to simplify the distribution of applications built using Eclipse. Unfortunately, we have not discovered a portable alternative to find out which directory an executable is running from.
